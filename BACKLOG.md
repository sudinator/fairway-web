- v1.111.2 - Fix duplicate-hole doubling on posted rounds (migrations 0076 + 0077, plus a client guard). Root cause: `holes` had no unique constraint on (round_id, hole_number), and both posting functions (post_game_rounds/post_group_rounds) do delete-then-insert, so two concurrent posts of the same (game,user) — likely in group scoring when several finish a tee group at once — each snapshot no committed holes under READ COMMITTED and both insert a full set, leaving 36 rows for 18. That doubled gross/net/Stableford and the scoring buckets and rendered each hole twice. Fix: 0076 adds unique index holes_round_hole_uk; 0077 changes both functions' hole insert to `on conflict (round_id, hole_number) do update` so a racing post updates in place instead of duplicating (validated on Postgres: double-post now yields 18 holes, gross correct; manual dup insert rejected). Client belt-and-suspenders: new dedupeHoles() in lib/golf.ts, applied at the round-load points in home.tsx and manage.tsx, collapses any duplicate hole rows (preferring the scored copy) so the UI can't double-count even on legacy data. Note: individual round delete is soft (deleted_at, holes survive); deleting a game with same-day rounds hard-deletes the round + holes.
